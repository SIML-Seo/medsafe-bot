## 0. 이 프롬프트의 사용 규칙 (먼저 읽어라)

- **불확실 표기 규율**: 아래 브리프의 API 스펙(오퍼레이션명·필드명·버전접미사)·일정·규제·PlayMCP/Widget 세부는 상당수가 웹검색 기반 *추정*이며 `⚠재확인` 표시가 있다. 이런 항목은 공식 문서/Swagger/실호출로 확정하기 전까지 코드에 **하드코딩하지 말고**, 아래 "검증 수단 가용성 게이트"를 따르라.
- **검증 수단 가용성 게이트 (핵심 — 정지 방지)**: 실호출/웹 접근이 불가능하면 각 `⚠재확인` 항목에 대해 **정지하지 말고** 다음을 하라.
  1. 그 값의 **공식 문서 URL**을 산출물에 명시한다.
  2. 그 값을 쓰는 코드를 `config/schema-map` **단일 파일 1곳**으로 격리해, 나중에 그 파일만 고치면 교체되도록 설계한다.
  3. §3.3 조인키처럼 갈림길이 있는 항목은 **양쪽 경로를 모두 스텁으로** 구현하고 런타임 플래그로 전환 가능하게 둔다.
  - 즉 "검증 불가"는 진행 정지 사유가 아니라 **격리 설계 사유**다.
- **fail-closed는 제품 철학이자 코딩 규칙**이다. 정의는 §6에 1회만 두고, 다른 섹션은 "§6 준수"로 참조한다.
- **작업 순서 강제**: 반드시 §9대로 **설계문서 먼저 → (자체 점검) → 구현 → 검증** 순으로 진행하라. 설계 없이 코드부터 쓰지 마라.

## 1. 역할 / 미션

### 맥락 (공모전)
- **대상 공모전**: 카카오 **AGENTIC PLAYER**(구 MCP Player 계열). ⚠재확인 — 최신 명칭·일정은 카카오 공식 공모전 페이지 및 PlayMCP 공지에서 확정.
- **일정(⚠재확인, 절대날짜 미확정)**: 예선 접수 마감·심사 리드타임·본선(이용자 투표, Widget UI 추가) 시점은 **모두 미확정**이다. 절대날짜는 §8.2 규칙에 따라 상대 일정으로 관리하고, 확정 날짜는 **사용자에게 확인 요청**하라.
- **채널**: 카카오 **PlayMCP**에 remote MCP 서버를 등록 → **Kakao Tools**(카카오톡 내 에이전트 실행 레이어)로 노출 → 최종 사용자는 **일반 카톡 유저**(하드웨어 없음).

### 미션
당신은 다음을 만든다:
- **remote MCP 서버 1개**. 사용자가 "지금 먹는 약들"(제품명/성분명)을 텍스트로 알려주면, **공식 공개데이터**(식약처 DUR·e약은요 등)로 **병용금기·중복성분·효능군중복·연령/노인/임부 주의**를 조회·판정하여 **🟢🟡🔴 신호등**으로 응답한다.
- 예선 = **채팅 자연어 응답 중심 MVP**. 본선 = **Widget UI 추가**.

### 최종 산출물 (요약 — 상세는 §10)
1. 설계문서 세트(아키텍처·tool 계약·데이터흐름·안전정책·매핑 파이프라인·재확인 결과).
2. 동작하는 MCP 서버 코드(TypeScript `@modelcontextprotocol/sdk` **또는** Python `FastMCP` — 이유와 함께 택1).
3. 로컬 매핑 마스터 DB 구축 스크립트(제품명↔코드) + 조인키 검증 노트.
4. 테스트 스위트(병용금기 회귀셋·매핑 정확도·fail-closed 동작·안전문구).
5. PlayMCP 등록 가이드 + 데모 시나리오.

## 2. 제품 개요

### 문제
노인 다제약물(polypharmacy), 만성질환자 다약제 복용, "감기약+진통제" 성분 중복 등 **일상 복약 실수**로 인한 위해. 기존 확인 수단은 웹/앱 직접 입력·회원가입이 필요 → 정작 위험군(노인·비전문가)에게 진입장벽.

### 타깃
1차: **부모/조부모의 복약을 대신 확인하는 자녀**(대리 확인). 2차: 여러 약을 동시 복용하며 스스로 점검하려는 일반인.

### 핵심 가치 & 차별화 (약끼리 대비 — 이 1블록으로 압축)
> **경쟁 선점 위협**: 약사가 만든 무료 **'약끼리'** 가 병용금기+동일성분+효능군중복+처방전 OCR을 **이미 웹으로 제공**. **"카톡 무설치 채널"만으로는 창의성 차별화가 약하다.** 약끼리(웹/앱 자기입력형)가 구조적으로 재현하기 어려운 축으로 차별화하라:
> 1. **대리 확인 흐름(caregiver flow)** — 자녀가 부모 대신, 설치·가입 없이 카톡 대화로. → *구현 함의*: `check_medication_safety.context`에 **"복용자≠사용자"** 및 연령/임부 필드.
> 2. **능동적 되묻기(clarification loop)** — 애매한 표현을 봇이 먼저 되물어 표준코드로 정규화. → *구현 함의*: tool 계약의 `AMBIGUOUS` 경로(§5.2).
> 3. **신호등 + 근거·출처·기준일** — 비전문가 가독성.
> 4. **에이전트 오케스트레이션 서사** — 여러 공식 소스를 한 대화로 자동 교차 조회 = MCP 컨테스트 스토리.
>
> 설계문서 첫 페이지에 **"약끼리 대비 하는 것 / 안 하는 것"** 표 1개를 넣어 창의성 심사를 정면 방어하라. (이 서사는 여기 1회로 끝내고 이후 섹션에서 반복하지 말 것.)

### 심사 3축 조준점
| 축 | 조준 | 방어 위치 |
|---|---|---|
| 창의성(파급력) | 카톡 네이티브 대리·대화형 복약안전 | 위 차별화 블록 |
| 편의성(UI/UX) | 신호등 요약 + 되묻기 + 본선 Widget | §7 |
| 안정성(데이터·보안) | 식약처 공식 데이터 + fail-closed + 면책 + 개인정보 무저장 | §6 |

**응답 래퍼(표준)**: `response > header(resultCode,resultMsg) + body(items > item[], numOfRows, pageNo, totalCount)`.

**응답 필드 접근 규칙 (필수)**: DUR은 `MIXTURE_ITEM_SEQ` 같은 SNAKE_UPPER, e약은요는 `itemSeq` 카멜 등 **소스마다 케이스 컨벤션이 다르고 XML/JSON에서 또 다를 수 있다**. 따라서:
- 필드를 **절대 하드코딩하지 말고**, 소스별 `FIELD_MAP`(논리명→실제응답키) 상수를 두어라.
- 실제 키 확정 전에는 **대소문자 무시(normalize) 조회**로 접근하고, 매칭 실패한 필드는 로그가 아니라 `unresolvedFields`로 수집해 **fail-closed로 연결**하라(§6).

### 3.3 코드체계 & 매핑 다리 (매핑 인프라의 핵심)

한국 의약품 코드 4축:
1. **품목기준코드(itemSeq)** — 식약처가 허가 품목마다 부여. DUR 품목/e약은요/낱알식별의 조인 키.
2. **주성분코드(9자리)** — 심평원 부여. **자릿수 배분은 문서로 확정 전 substring 슬라이싱 금지**(§4.2(4) 참조). 성분 축 그룹핑에 사용.
3. **의약품 표준코드(KD/바코드 13자리)** — 유통단위. 바코드 입력 시.
4. **ATC코드(7자리 5단계)** — WHO 분류. 상위 레벨 일치로 효능군중복 근사 판정.

**매핑 마스터(로컬 조인용, MVP 핵심)**:
- **심평원 ATC코드 매핑목록** — 데이터셋 ID **15118958**(파일데이터/CSV, 월갱신, 약 21,953행). 컬럼: 식약분류 / **주성분코드(9자리)** / **제품코드(9자리)** / 제품명 / 업체명 / ATC코드 / ATC코드명칭. → 이 한 파일로 제품명↔주성분코드↔제품코드↔ATC 오프라인 조인.
- 심평원 **약가마스터_의약품표준코드** — 데이터셋 ID **15067462**(월갱신). 표준코드(KD)↔제품↔주성분코드. 바코드 지원 시.
- 심평원 **DUR 의약품 목록** — 데이터셋 ID **15127983**(CSV, 월갱신, 실시간 아님, 배치 다운로드용).

> **⚠최중요 리스크 — 반드시 실데이터 조인으로 검증**: ATC매핑목록의 **'제품코드(9자리)'가 DUR/e약은요의 품목기준코드(itemSeq)와 동일 키인지**, 아니면 심평원 자체 제품코드라 별도 변환이 필요한지 불명확. **셋이 100% 동일 키로 조인되지 않으면 매핑 파이프라인 전체가 무너진다.**
> 검증 불가 시 §0 게이트대로 **양쪽 경로(동일키 가정 / 별도 변환)를 모두 스텁으로 구현 + 런타임 플래그 전환**하고, 어느 쪽인지는 조인키 검증 노트에 미해결로 남겨라.

## 4. 핵심 난제 — 표현→표준코드 매핑

> **진짜 어려운 부분은 판정 API 호출이 아니라, 사용자가 말하는 '제품명/브랜드/성분/색깔'을 DUR이 요구하는 `itemSeq`/주성분코드로 변환하는 것**이다(오타·구어·브랜드↔성분 혼용). 아래 규칙을 그대로 구현하라.

### 4.1 매핑 파이프라인

```
사용자 발화
   │
   ├─ [입력 분류기] 제품명 like  vs  성분명 like  (성분사전 매칭으로 먼저 판별)
   │
   ├─ (a) 제품명 경로  → 로컬 마스터 fuzzy 검색 → itemSeq 후보
   ├─ (a') 성분명 경로 → 주성분코드 직접 매핑(제품 특정 시도 안 함) → 성분축 판정 전용
   ├─ (b) 낱알(색/모양/각인) 텍스트 → 낱알식별 조회 → itemSeq 후보  [본선, 되묻기 필수]
   └─ (c) 처방전/약봉투 사진·OCR → **MVP 완전 제외**(입력 스키마에 이미지 필드 없음)
   │
   ▼
[매핑 신뢰도 판정 — §4.2]
   ├─ 단일 고신뢰 후보 → CONFIRMED (단, 판정 전 사용자에게 매핑 결과 되보여 확인)
   ├─ 복수/저신뢰      → AMBIGUOUS → ★되묻기(임의 선택 금지)
   └─ 후보 없음        → NOT_FOUND → fail-closed: 🟡 "특정 불가, 약사 확인 권장"
   │
   ▼
확정 itemSeq(및/또는 성분코드) 리스트
   ├─ DUR 8종 교차 판정(§4.3 알고리즘) → 신호등
   ├─ 주성분코드/ATC 상위 일치 → 중복성분/효능군중복 선판정
   └─ e약은요 → 사용자용 설명(가능 품목만)
```

### 4.2 구체 설계 규칙

**(1) 로컬 마스터 DB (SQLite 권장)** — ATC매핑목록(15118958) 적재. `제품명 / 주성분코드 / 제품코드 / 업체명 / ATC / ATC명칭` 인덱스. 제품명 정규화 컬럼 추가. **⚠먼저 §3.3 조인키 검증 후 스키마 확정**(미검증 시 §0 게이트).

**(2) 한글 매칭 규칙 (구체 — 라틴 가정 금지)**
- **정규화**: 공백/괄호/제조사/규격(mg,밀리그람)/제형어(정,캡슐,정제,서방정,현탁액) 분리 → 코어 브랜드·성분 토큰 추출.
- **유사도**: 토큰셋 비율 + 초성 일치 보너스 + 자모 분해 편집거리 병용(순수 라틴 편집거리 단독 금지).
- **임계값 τ(초기값, §9 Step3 회귀셋으로 튜닝 후 최종값 문서화)**:
  - 정규화 완전일치 → `CONFIRMED`
  - 토큰셋 ≥ 0.90 & 후보 1개 → `CONFIRMED`
  - 0.75 ~ 0.90, 또는 후보 복수 → `AMBIGUOUS`(되묻기)
  - < 0.75 → `NOT_FOUND`(fail-closed 🟡)
- 동일 성분 다수 제네릭·다규격으로 후보 폭증 시 되묻기로 흡수.

**(3) 구어·별칭 처리 (별도 alias 테이블 필수)**
- ATC매핑목록에는 **정식 제품명만** 있어 "판콜"(→판콜에이/에스), 축약·슬랭·성분속어는 마스터만으로 NOT_FOUND가 된다. → **별칭(alias) 테이블을 마스터와 별도로 수기 구축**(데모·상용 상위 100~200 OTC의 구어·축약·성분별칭 → 대표 itemSeq/성분).
- 별칭 테이블 없이는 구어 입력이 대부분 되묻기로 빠진다는 점을 로드맵에 명시.

**(4) 성분 축 선판정 (슬라이싱 매직넘버 금지)**
- **주성분코드 9자리의 자릿수 배분(어느 구간이 성분식별자인지)을 심평원 코드체계 문서로 확정하기 전에는 substring 슬라이싱 금지.** 임의 자르기는 중복성분 오탐/누락에 직결된다.
- 확정 전 기본 판정은 **ATC매핑목록의 '주성분코드' 전체값 동일 + ATC 4~5레벨 일치**로 중복성분/효능군 근사. 부분 슬라이싱은 자릿수 확정 후 도입하고, **슬라이싱 폭은 매직넘버 금지·확정 출처를 주석에 남긴다**(주석은 self-contained).

**(5) 되묻기(disambiguation) UX**
- 후보 최대 5개를 버튼/카드로 제시, 사용자가 고르게 함. **절대 임의 선택 금지.**
- 확정 후 "이 약이 맞나요? [○○정 500mg / □□제약]" 되보여 오매핑을 사용자가 잡게 함(→ 미탐뿐 아니라 **오탐(오매핑) 방지 장치**, §6-C 참조).

### 4.3 병용금기 판정 알고리즘 (핵심 — 통째로 빠뜨리지 말 것)
> DUR 병용금기 API는 대체로 **약 1개를 넣으면 그 약의 '금기 상대 목록'을 반환**하는 구조로 알려져 있다(⚠Swagger로 확정). **API에 약쌍을 직접 던진다고 가정하지 마라.**
- 알고리즘: **각 itemSeq마다 병용금기 API 호출 → 응답의 '금기 상대 itemSeq/성분 목록' 수집 → 내 복약목록의 나머지 itemSeq/성분과 로컬 교집합.** N개 약이면 N회 호출 + O(N²) 로컬 대조.
- 연령/노인/임부/용량/기간/서방정 등 나머지 유형도 동일하게 "약별 조회 → 로컬 판정" 패턴. 실제 파라미터/응답 구조는 §3.1 Swagger로 확정.

**(페이지네이션 — 미탐 방지 필수)**: `numOfRows` 기본3(e약은요), DUR도 페이징. 금기목록이 잘리면 🟢 오탐이 나온다(fail-closed 위반). → 판정 소스는 `totalCount`를 확인해 **전 페이지 수집**, `numOfRows`를 충분히(예 100) 키워라. **페이지 수집 실패/중단 시 해당 약은 fail-closed(🟡).**

### 4.4 2주 MVP 범위 축소안 (강제)
| 입력 경로 | MVP(예선) | 본선/확장 |
|---|---|---|
| **텍스트 제품/성분명 입력** | ✅ **1차 범위** | — |
| 낱알(색/모양/각인) 텍스트 조회 | 선택(여유 시) | ✅(되묻기 필수) |
| 처방전/약봉투 **OCR·이미지** | ❌ **완전 제외**(입력 스키마에 이미지 필드 없음) | 후보(넣으면 개인정보 무저장·즉시 파기) |
| 사진→약 **자동 이미지 식별(딥러닝)** | ❌ 제외 | 후보(비현실적) |

> MVP는 **텍스트 입력 + 되묻기 + DUR 교차판정 + 신호등**으로 완결. 나머지는 "본선 확장" 로드맵으로만.

## 5. MCP 서버 아키텍처 & tool 계약

### 5.1 표준 & 제약 (준수)
- MCP 서버는 `tools/list` + `tools/call`(JSON-RPC)로 tool 노출. tool 정의 = `name`/`title`/`description`/`inputSchema`(JSON Schema, 필수)/`outputSchema`(선택)/`annotations`(선택).
- `tools/call` 응답 = `content`(배열) + 선택적 `structuredContent` + `isError`(bool).
- **에러 2경로**: (1) 프로토콜 에러 = JSON-RPC `error`(예 알 수 없는 tool `-32602`), (2) 실행 에러 = `result` 안 `isError:true` + `content` 에러 텍스트. **판정 실패(API 타임아웃/데이터 없음)는 `isError:true`로 안전하게 노출.**
- **stateless / 능동푸시 불가**: stateless HTTP MCP는 세션 추적이 없어 **서버→클라 능동 푸시(알림·리마인더) 불가**. 복약 리마인더 등은 MCP 채널 밖에서 별도 구현. **MCP tool은 "요청에 응답"만 하도록 설계.**
- **transport**: remote 배포는 **Streamable HTTP**(단일 `/mcp` 엔드포인트) 표준. 구형 SSE는 하위호환.
- **SDK 택1**: TS `@modelcontextprotocol/sdk`(`McpServer`+`registerTool`+`StreamableHTTPServerTransport`) **또는** Python `FastMCP`. 선택 이유를 설계문서에 명시.
- **read-only 명시**: 모든 tool에 `annotations.readOnlyHint=true`(안전심사·human-in-loop 완화에 유리). tool `name`은 안정 고정, 버전은 서버 메타로.
- **⚠MCP 스펙 세부는 지식컷 이후일 수 있음 — 타깃 LLM이 `modelcontextprotocol.io` 공식문서로 재확인.** 열 수 없으면 **"요청-응답 tool + Streamable HTTP"라는 최소 안전 가정만** 쓰고, 알림/푸시/세션 의존 기능은 설계에서 배제(`[가정]` 태그).

### 5.2 tool 계약 (초안 — 확정하라)

### 5.3 판정→신호등 매핑 규칙 (fail-closed 내장 — §6 준수)
| 상황 | verdict | 표시 |
|---|---|---|
| DUR 병용금기/임부·연령 금기 히트 | `WARN` | 🔴 |
| 중복성분·효능군중복·노인/용량/기간 주의 히트 | `CAUTION` | 🟡 |
| 매핑 불확실·API 무응답·항목 특정 실패·**8종 중 일부 유형 조회 실패** | `UNCERTAIN` | 🟡 (**절대 🟢 아님**) |
| 모든 약 확정 + DUR 조회 성공 + 금기/주의 **미조회** | `NO_KNOWN_FINDINGS` | 🟢 (**문구는 "등록된 금기 미조회"; "안전" 금지**) |

> **부분 판정 규칙**: 8종 중 일부만 성공하면 `verdict=UNCERTAIN`(🟡) + 성공 유형만 `findings` + 실패 유형은 `failedTypes`로 노출. `context`가 `unknown`이라 연령/임부 판정 불가면 그 유형을 숨기지 말고 "판정 보류" finding으로 노출.
> **신호등 요약 텍스트는 서버가 템플릿으로 생성**해 `content[text]`에 넣어라(호출 LLM 자유 서술 금지 — 금칙어 누출 방지). `structuredContent`는 기계용 JSON. **둘은 표현이 다르며, JSON을 text로 그대로 이중 직렬화하는 것은 선택**이다(PlayMCP/Kakao Tools가 structuredContent를 소비하는지 확인 후 결정). 예:
```json
{ "result": {
  "content": [{"type":"text","text":"🟡 주의 1건 · 🔴 금기 0건 …(서버 생성 요약 + 디스클레이머)"}],
  "structuredContent": {"verdict":"CAUTION","findings":[{"type":"DUP_INGREDIENT","origin":"LOCAL_INGREDIENT","level":"YELLOW","a":"타이레놀정 500mg","b":"게보린정","reason":"아세트아미노펜 중복","source":"주성분코드","baseDate":"2026-07-01"}],"unresolved":[],"checkedTypes":["USJNT_TABOO","DUP_INGREDIENT"],"failedTypes":[]},
  "isError": false } }
```

## 6. 안전 / 법적 요구사항 (안정성 축 통과 조건 — fail-closed 단일 정의)

> **법적 포지셔닝(방어 근거)**: 식약처 '의료기기와 개인용 건강관리(웰니스) 제품 판단기준' 지침상 **'정보 안내/제공'은 웰니스(비의료기기)**, **'진단·치료·처치·예방' 임상목적이면 의료기기**. 선례로 유사 복약확인 앱이 허가 없이 운영 중. **이 봇은 반드시 '공개데이터 조회·안내'로 프레이밍**하고, '판정/진단'이 아니라 '주의 정보 등급'으로 명명하라. ⚠타깃은 최신 식약처 웰니스 지침 원문에서 '의약품/복약 정보제공 소프트웨어' 예시를 재확인.

### 안전 규칙 (프롬프트/코드에 그대로 박아라)
1. **진단·처방·치료지시 금지**: "무슨 병이다 / 이 약 드세요·끊으세요 / 용량 바꾸세요"류 생성 금지. 오직 **이미 보유·처방된 약들 사이의 공개 안전정보 조회·요약**만.
2. **fail-closed 원칙(★단일 정의)**: 매핑 불확실 / DUR 응답 공백 / 약 특정 실패 / 페이지 수집 실패 / 8종 중 일부 실패 / 필드 매칭 실패(`unresolvedFields`) → **절대 🟢 아님**, 🟡로 강등 + "약사·의사 확인 권장". **"데이터에 병용금기 없음" ≠ "안전함"** → **"등록된 병용금기는 조회되지 않았습니다"** 로 *사실만* 진술.
3. **오매핑(false positive) 위해 방지**: 🔴/🟡 판정에 **사용된 실제 매핑 제품명·성분을 반드시 되보이고**, 🔴 카드에 **"이 약이 아니면 이 경고는 무시하세요"** 부착. 자동확정(CONFIRMED)이라도 판정 결과 제시 전 매핑 확인을 기본값으로. 되묻기는 미탐뿐 아니라 **오탐 방지 장치**.
4. **자가 중단·변경 조장 금지**: 각 finding에 **"무엇을 하라가 아니라, 무엇이 조회됐다"는 사실-프레이밍** 문구 고정. 🔴/🟡 공통으로 **"이미 처방받은 조합일 수 있으며 임의 중단 금지, 처방의·약사에게 문의"** 부착. "먹지 마세요"류 임상 단정 금지.
5. **스코프 한계 능동 고지**: DUR은 의약품 코드 기반이라 **건기식·식품·한약 상호작용은 구조적으로 0**이다. 입력에 비의약품(오메가3·자몽·홍삼 등) 의심 토큰이 있으면 **"본 서비스는 의약품 간 정보만 조회 가능"**을 명시(`check_medication_safety.disclaimer` 상수에 포함).
6. **응급 신호 최우선**: 아나필락시스·호흡곤란·의식저하·심한 흉통 등 언급 시 **즉시 "119/응급실"** 안내를 최우선 출력, 상호작용 조회로 뭉개지 말 것.
7. **오남용·불법 유도 차단**: 마약류·오남용우려약·사후피임약·향정약에 대한 "더 먹어도 되나 / 구하는 법 / 처방 우회"류 응답 금지. **특정 약국·병원 유인/알선/구매링크 금지**(약사법·의료법 경계).
8. **의료기기 회피 프레이밍**: "중복 성분이 발견되었습니다(정보 제공)" = OK / "이 약 먹으면 위험합니다(임상 단정)" = 지양. 신호등은 '위험도 판정'이 아니라 **'주의 정보 등급'**.
9. **금칙어(서버단 강제 필터)**: **"안전합니다" 단정 금지어 등록.** 대신 "등록된 금기는 조회되지 않았습니다" + "단, 이 결과는 처방약/OTC/건강기능식품/한약 일부를 포함하지 못할 수 있으니 약사 확인을 권장합니다" 강제 삽입. 안전 문구를 호출 LLM 재량에 맡기지 말 것.
10. **표준 디스클레이머(매 응답 하단 고정, 서버 주입)**:
    > *본 정보는 식약처 공개데이터 기반 일반 참고용이며 의사·약사의 진단·처방·복약지도를 대체하지 않습니다. 실제 복용·중단·변경은 반드시 의사 또는 약사와 상담하세요. 응급 증상 시 즉시 119.*
    + 최초 온보딩 시 **"본 서비스는 의료기기가 아닙니다"** 고지 1회.

### 보안 / 개인정보
- **개인정보 최소·무저장**: 복약 목록을 **서버 미저장·처리 후 즉시 파기**. 세션 내 복약목록은 프로세스 메모리에만 두고 TTL 후 폐기. 로그는 **익명 집계(건수)만** — 개인식별정보·질병정보 미로깅.
- **카카오 OAuth 사용자 식별자**: 판정에 불필요하면 **수신·보존 금지**. 데이터 국외이전·개인정보처리방침 필요 여부를 PlayMCP 등록 요건과 함께 확인 항목에 포함.
- **시크릿**: `serviceKey` 등은 환경변수/시크릿 매니저. 코드·저장소 하드코딩 금지. serviceKey는 **Decoding 키를 코드에 넣고 HTTP 클라이언트가 자동 인코딩**하게 하라(수동 `encodeURIComponent` **이중 인코딩 금지** — `%2B`→`%252B`, SERVICE_KEY_IS_NOT_REGISTERED_ERROR 방지). **부팅 시 1회 헬스체크 호출로 `resultCode` 정상(예 `00`)을 확인하는 self-test**를 넣어라.
- (본선에서 OCR/이미지 도입 시에만) 민감정보 처리 동의·수집최소화·즉시 파기 설계. ⚠개인정보보호법·카카오 헬스/의료 콘텐츠 정책·PlayMCP 심사 요건 재확인.

### 7.2 본선 — Widget UI 설계 개념
> **⚠Widget 실체 불명**: 본선 Widget이 (a)기존 카톡 챗봇 카드(basicCard/listCard/carousel) 재사용인지, (b)MCP-UI(embedded HTML resource)인지, (c)Kakao Tools 전용 신규 Widget JSON인지 **공식 미공개**. 카카오가 본선 진출자에게 배포하는 스펙으로 확정하라. `structuredContent`→Widget 자동매핑 여부도 ⚠확인.

- **스펙 미공개 상태에서는** 매핑표를 **'카톡 카드 3형(listCard/carousel/basicCard) 가정 버전'으로만** 작성하고 `[가정]` 태그. `structuredContent`를 **렌더-중립적(표현 무관 데이터)**으로 설계하는 데 집중하고, 실제 Widget JSON은 본선 스펙 수령 후 붙인다.
- 카드 어휘(개념적 상한, ⚠재확인): **listCard**(헤더 "복약 안전 점검 결과", 행=약쌍+🔴/🟡/🟢, 최대 5), **carousel**(약별 카드, 최대 10), **buttons**(`webLink` 상세 출처, 라벨 14자), **quickReplies**(최대 10).
- 설계문서에 **"structuredContent 필드 → Widget 요소" 매핑 표**(가정 버전)를 만들어 두면 본선 스펙 확정 시 즉시 붙일 수 있다. **Widget은 `structuredContent`를 단일 진실원천으로 렌더**하도록 tool을 설계하라.

## 8. 범위 구분 & 타이밍

### 8.1 예선 MVP vs 본선 (범위 현실화)
| 항목 | 예선 MVP | 본선 |
|---|---|---|
| 입력 | 텍스트 제품/성분명 | + 낱알 텍스트 / (검토)OCR |
| 판정 | **병용금기 + 중복성분(성분코드) 2종 필수**, 나머지 6종 best-effort | 8종 + 정교화 |
| 매핑 | 로컬 마스터 fuzzy + alias + 되묻기 | + 낱알/바코드 |
| 설명 | e약은요(가능 품목) | 동일 |
| UI | 채팅 신호등 응답 | **Widget**(가정 버전) |
| 알림 | 없음(MCP는 요청-응답만) | 리마인더는 MCP 밖 채널 |

> **공수 리스크 경고(2주 현실성)**: 마스터DB 적재·정규화·alias 테이블·한글 퍼지매칭 튜닝만으로도 며칠이 든다. 8 API 전 필드 확정·파싱·판정매핑을 2주에 다 넣는 것은 위험. → **최소 데모 경로(happy path)** = *'OTC 상위 N개 alias + 병용금기·중복성분 2종 + 텍스트입력'*을 먼저 end-to-end로 완성하고 데모 시나리오를 여기에 고정. 나머지 유형·낱알·ATC효능군은 시간 남으면. (수용기준 §10도 이 축소에 맞춰 '핵심 2종 필수 + 나머지 best-effort'.)

### 8.2 타이밍 규칙 (절대날짜 미확정)
```
절대날짜(예선 마감·심사 리드타임·본선 일정) = 미확정 → 사용자에게 확인 요청.
그 전까지는 상대 규칙만 사용:
  개발 완료 목표일 = 등록 마감 D-5
  └ serviceKey 활용신청(자동승인이나 운영계정 상향은 시간 소요 가능)
  └ PlayMCP 서버 등록 + (있다면) 카카오 심사/승인 소요  ⚠재확인
  → "코드 완성 ≠ 제출 완료". 등록·심사 큐 진입까지 역산해 버퍼 확보.
```
> ⚠재확인: PlayMCP 공개 전 카카오 심사 프로세스·기준·소요시간, 등록 폼 요구 필드/허용 transport(Streamable HTTP만인지 SSE 허용인지)/무인증 공개 tool 허용 여부, OAuth 사용자 컨텍스트를 tool 내부에서 받을 수 있는지. **PlayMCP는 이용자 카카오 OAuth 기반이라 최초 1회 사람이 OTT→토큰 교환·주기 갱신 수동**일 수 있음. **위 세부는 타깃 LLM이 공식문서로 재검증.**

## 9. 작업 프로세스 (반드시 이 순서)

### Step 1 — 설계문서 작성 (코드 이전, 필수)
아래를 문서로 먼저 산출하고 **자체 점검** 후 구현으로 넘어가라(셀프 리뷰만으로 확정하지 말고, 검증 결과·미해결을 명시):
1. **아키텍처 다이어그램**: 사용자→MCP tool→(로컬 마스터/DUR/e약은요)→응답. stateless 제약 명시.
2. **tool 계약 확정본**: §5.2를 실제 스키마로 확정(입력/출력/에러/불확실/방어적 재검증).
3. **매핑 파이프라인 상세**: §4 — 입력 분류기, τ 값, alias 테이블, 성분/ATC 선판정, 병용금기 알고리즘(§4.3), 페이지네이션.
4. **데이터 흐름 & 조인키 검증 결과**: §3.3 `제품코드=itemSeq?` 실조인 결과(또는 §0 게이트에 따른 이중경로 스텁 + 미해결 표기).
5. **안전정책**: §6 체크리스트를 시스템 프롬프트/서버 코드 규칙으로 번역(금칙어 필터·디스클레이머 주입 위치 포함).
6. **차별점 방어표**(약끼리 대비, §2).
7. **불확실·재확인 목록**: 부록의 각 항목을 "확인됨 / 미해결(사유·격리 위치)"로 정리.

> **⚠재확인 우선순위(구현 전)**: ① §3.3 조인키(제품코드=itemSeq) ② DUR 8종 오퍼레이션명·필드명(Swagger) ③ 라이선스/트래픽 ④ PlayMCP 등록·심사·transport. **접근 불가 시 각 항목 §0 게이트로 격리(정지 금지).**

### Step 2 — 구현
- 로컬 마스터 DB 적재 스크립트 + alias 테이블 → 입력 분류기·매핑 모듈 → DUR/e약은요 클라이언트(`FIELD_MAP`·재시도·타임아웃·페이지 수집·`isError` 매핑·serviceKey self-test) → tool 핸들러(방어적 재검증) → **서버측 신호등 포맷터·금칙어 필터·디스클레이머 주입** → 캐싱(itemSeq별 응답 TTL 캐시로 호출 절감·한도 근접 시 degrade).
- 시크릿 환경변수. 개인정보 무저장. 코드 주석은 self-contained(외부 문서 참조 금지).

### Step 3 — 검증 (테스트 필수)
1. **병용금기 회귀셋**: 알려진 금기쌍(⚠실제 조합은 DUR 데이터로 확정) N건 → 🔴. 알려진 무금기쌍 → 🟢(단 "미조회" 문구).
2. **중복성분 케이스**: 아세트아미노펜 함유 2제품 → 🟡 중복 감지.
3. **매핑 정확도**: 정확명/오타/구어(alias)/성분명/다규격 입력 세트로 CONFIRMED/AMBIGUOUS/NOT_FOUND 분류 정확도 측정.
4. **fail-closed 동작**: API 타임아웃·빈 응답·페이지 잘림·미매핑·8종 부분실패·필드 매칭 실패 → 반드시 🟡 + 약사확인, 🟢 안 나옴. `isError` 경로 정상 노출.
5. **오탐·안전 문구 검사**: 오매핑 시 "이 약이 아니면 무시" 부착, "안전합니다" 금칙어 미출현, 디스클레이머·스코프 문구 상시, 응급어 우선 응답, "임의 중단 금지" 부착.

> 임의 가정은 `[가정]`으로 표기. 불확실은 "안전합니다"류로 얼버무리지 말고 fail-closed로 처리하라.

## 10. 산출물 목록 & 수용 기준

### 산출물
1. `design/` — 아키텍처·tool 계약·매핑 파이프라인·데이터흐름·안전정책·차별점 방어표·재확인 결과.

4. 테스트 스위트(§9 Step3 5종).
5. `README` — 로컬 실행·serviceKey 설정(env)·PlayMCP 등록 절차·데모 시나리오.

### 수용 기준 (Acceptance)
- [ ] `resolve_medications`가 정확명/오타/구어(alias)/성분명/다규격 입력에 대해 CONFIRMED/AMBIGUOUS/NOT_FOUND와 `inputKind`를 올바르게 분류하고, 모호 시 후보를 반환한다.
- [ ] `check_medication_safety`가 **병용금기·중복성분(핵심 2종)을 필수로**, 나머지 유형을 best-effort로 교차 조회하고, 중복성분/효능군을 성분·ATC로 선판정하며(`origin` 구분), `verdict`+`findings`(출처·기준일)+`checkedTypes`/`failedTypes`를 `structuredContent`로 반환한다.
- [ ] **호출자 불신 방어**: 미확정/무효 itemSeq를 서버가 재검증해 `unresolved`로 밀고, 되묻기를 스킵당해도 🟢가 나오지 않는다.
- [ ] **fail-closed 100%**: 매핑 실패/API 무응답/페이지 잘림/부분실패/특정 불가에서 🟢가 절대 안 나오고 🟡+약사확인이 나온다. "안전합니다" 금칙어가 어디에도 없다(서버 필터).
- [ ] **오매핑 방어**: 🔴/🟡에 되보인 매핑명 + "이 약이 아니면 무시" + "임의 중단 금지"가 부착된다.
- [ ] 모든 응답에 출처·기준일·표준 디스클레이머·스코프 문구가 부착되고, 최초 온보딩 시 "의료기기 아님" 고지가 1회 나온다. 응급 발화에 119가 최우선 출력된다.
- [ ] 신호등 요약·디스클레이머·금칙어 필터가 **서버단**에서 생성·강제된다(호출 LLM 자유서술 아님).
- [ ] §3.3 조인키 검증 결과가 문서화(또는 이중경로 스텁+미해결 표기)되어 있고, 불일치 시 변환 로직이 구현되어 있다.
- [ ] DUR 오퍼레이션명·필드명이 Swagger 실확인으로 교정(또는 `config`/`FIELD_MAP`로 격리+문서 URL 명시)되어 있고, 추정→확정 여부가 표기되어 있다.
- [ ] 개인정보 무저장(약 목록 미영속, 민감정보 미로깅, OAuth 식별자 미보존)이 코드로 확인된다.
- [ ] MCP 서버가 Streamable HTTP로 기동되고 `tools/list`·`tools/call`이 스펙대로(에러 2경로 포함) 동작한다. serviceKey self-test가 부팅 시 통과한다.
- [ ] 재확인 목록 각 항목이 "확인됨/미해결(사유·격리 위치)"로 정리되어 있다.
